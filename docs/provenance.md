# Artifact provenance

The repository was reconstructed from retained simulation material on 20 July 2026.

OpenAI Codex assisted with repository preparation by collating retained material, recovering and inspecting configuration, documenting provenance, adding validation, and reconstructing missing convenience wrappers from the evidenced setup. Codex did not create the original simulation artefacts, synthetic source data, research scenarios, forensic datasets, or published datasets. Any newly reconstructed wrapper is identified as repository-preparation work rather than an original recovered script.

OpenWolf was used for context management during this preparation work, reducing repeated context loading and associated token usage. OpenWolf did not create the original research, simulations, or artefacts.

- The Windows lifecycle scripts and synthetic Active Directory fixtures came from the copied `Simulation scripts/Win scenario` directory. The original shutdown script had unmatched quoting and could not execute; it was replaced by the dry-run-first `vm-session-cycle.sh`.
- `data/windows/ncsc-pwned-passwords-top-1000.txt` is the exact first-1,000-entry subset of the 100,000-line NCSC `PwnedPasswordsTop100k` file retained in the simulation host's `Documents` directory. This subset drove the RDP brute-force password attempts. Subset SHA-256: `ffaed15c35523a8a0225d505f6c13bb1d3325957ddc9aa48d7055b1d2eebd31b`.
- The Windows scanner fixtures were recovered from the simulation host's `Documents` directory. The 4,666-row benign RDP host and gateway files are positionally paired and were used to model public scanner identities on the isolated `rdpscan` network. SHA-256: hosts `2569df685a86e8066d32f47a860f45c9299eaa5a884aaaeeb51f860f7c37ebc9`; gateways `4199139aa38782b8c3880710f6178617a1328f52d0e0c1f6ef54f6ab45987301`. The 3,056-row malicious RDP pool, SHA-256 `30a0aaf8395d58abbd83f31d7c55cc9f083c36f8a71151fde0d1fc4bfc0d3ded`, was retained for the attack-side source selection. These are historical GreyNoise-derived identities, not synthetic personal data, and are included only as simulation artefacts.
- The e-commerce CSV fixtures came from the copied `Simulation scripts/ecommerce scenario` directory. All personal, account, credential, contact, payment-card, and transaction values are synthetic data created using GenerateData.com; they do not represent real people. The scanner IP fixtures are historical GreyNoise/IP geolocation source identities used only inside the controlled simulation.
- The complete e-commerce database is `data/ecommerce/ecomm_updated_Luhn_minimal_sales-backup_20240113.sql`, recovered from the simulation host's `Documents/Staging` directory. It is a phpMyAdmin dump of the MariaDB `ecomm` database generated on 13 January 2024 and includes the full schema, products, sales and 1,003 users. The 1,000-address `web-shop-users.csv` JMeter fixture matches the 1,000 generated database accounts; the dump also contains three original synthetic application accounts. Source SHA-256: `76351572fa9a8dc48a6ca994c7f98445d58a066641770f1c528a43eb008e2339`.
- The base storefront is identified as [e-Commerce Site using PHP/PDO, PHPMailer, ReCaptcha, and PayPal](https://www.sourcecodester.com/php/12287/e-commerce-site-using-php-pdo.html), submitted by `nurhodelta_17` and mirrored at [`oretnom23/PHP-eCommerce-Site`](https://github.com/oretnom23/PHP-eCommerce-Site). Identification is supported by the exact distinctive route set used by the retained JMeter plan (`cart_add.php`, `cart_view.php`, `category.php`, `product.php`, `profile.php`, `sales.php`, and `verify.php`) together with the matching `ecomm` schema, products, user-registration fields, cart and sales tables.
- The host's 13 January 2024 web-root backup contains a locally modified `sales.php` (SHA-256 `3c476121928ada7acb10e7b2a01df46959cb0ac58cb5cd11fc384b5847c0c72e`). Comparison with `oretnom23/PHP-eCommerce-Site` commit `59d085a9accbda883ae718c017a4cd77cbdad549` isolates a four-line checkout modification: the PayPal callback condition and supplied payment ID are replaced by a 90%-success random branch and a generated `PAYID-FAKE-XQWEDES...` identifier. The success branch performs the original local sale inserts and clears the cart; the failure branch performs no sale update and both paths redirect to `profile.php`. The retained JMeter plan calls `/sales.php` directly. `scenarios/ecommerce/site-modifications/offline-paypal-random-outcome.patch` reproduces only this local change without bundling the third-party storefront.
- [Damn Vulnerable Web Application (DVWA)](https://github.com/digininja/DVWA) was a separate, intentionally vulnerable component added to the e-commerce host on 28 May 2024. The retained host checkout has Git origin `https://github.com/digininja/DVWA.git`; DVWA is GPL-3.0-or-later and is not bundled in this repository.
- `user-journeys.jmx` is based on the later 14 February 2024 host copy. Unlike the older copied plan, it targets port 80, enables random UK source-IP binding, and contains the final weekly load profile. Its seven Ultimate Thread Group rows represent the days of the week; per-row initial delay, startup, hold, shutdown, and user-count values vary demand within each day. The original plan comment records a 03:00 start alignment, with sequence day zero beginning on Wednesday and the first displayed row corresponding to Thursday. Machine-specific paths and target values were parameterised only in the cleaned copy.
- CMU Ghosts v7 `timeline.json` and `application.json` for the ordinary office clients were recovered read-only from the retained `Win10-Ghosts-3` VDI. The installed Ghosts directory had been moved to the Windows recycle bin, but its NTFS records and content remained recoverable. Only configuration is included; CMU binaries and stock content are not redistributed.
- The separate administrator-work-from-home configuration was recovered read-only from the final February 2024 `Win10-Ghosts-0` VDI in the original `OWC-SSD/VirtualBox VMs/Windows Office Network Simulation` tree. The original tree and retained backup have the same 79,918,268,416-byte disk size, modification timestamp, VDI image UUID and VDI modification UUID. Its active timeline contains a GHOSTS `Rdp` handler targeting `10.44.0.12` from 08:45 to 17:15 with the `admin` entry in `C:\creds.txt`; the referenced synthetic credential file was recovered from the same disk. Its password field is Base64-encoded rather than encrypted and decodes to the synthetic administrator password used in the simulation, as confirmed by the researcher. Strict-JSON copies under `scenarios/windows/ghosts/admin-config/` remove one trailing comma from the timeline and one from the credential file. Exact recovered SHA-256 values are: timeline `e188128dbaff1874b8230e21276a4ee1d44f1ff707b75783318b95fb02e04fb1`, application `7cb0a12d760729091fc65b0f1bdf6e395779dd5f279bc674ab0d294e11ef8bd5`, and credentials `8ab11fec135c2a4eb83467a5bfac01bd8bea74dc48189d41319ed230fbdba97b`.
- Dedicated historical Nmap and HTTP-scan VM names were present, but no standalone scanner shell scripts were found outside their disks. The included scanner wrappers reconstruct the documented behaviour and default to printing commands without executing them. The Windows VM metadata and DNS records consistently identify `10.44.0.12` as the server address; the retained RDP-scanner VM metadata shows the benign host-pool addresses loaded as aliases on `eth0`.
- Attacker activity was mostly manual. The researcher interactively initiated and controlled the intrusion sequence; scripts and security tools automated only bounded operations such as RDP password attempts, background Nmap traffic, SQLmap, and web scans. The repository therefore documents a human-led scenario rather than an autonomous attack agent.
- The original scenario record states that the RDP password attempts began with Hydra and were manually switched to Crowbar on 4 February 2024. Server-side forensic evidence confirms the high-volume attempt stream and supports the recorded switch timing, but cannot identify the remote client executable; Hydra and Crowbar are therefore documented as provenance claims rather than host-forensically verified tool attribution.

## Historical originals

`archive/original/` preserves the unmodified lifecycle scripts, all three recovered JMeter plan variants, the host's bulk IP-alias notes, and the byte-for-byte February 2024 administrator GHOSTS configuration. These files intentionally retain host paths, historical credentials, disabled components, and known defects because they document the actual setup. Use the cleaned scripts and strict-JSON configuration under `scenarios/` for reproduction.
